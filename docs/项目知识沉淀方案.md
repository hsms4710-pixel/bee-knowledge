# Graph-Driven Development 项目知识沉淀方案

> 基于《洛克王国：世界》AI 增效实践，为本项目设计的 MapReduce 知识沉淀系统

## 一、设计理念

### 1.1 核心思想

```
┌─────────────────────────────────────────────────────────────────────────┐
│  AI 的边界之外，仍然是人的判断、人的责任、人的创造力。                       │
│  Think what AI can't. Build what AI can.                                │
└─────────────────────────────────────────────────────────────────────────┘
```

**AI 的特点：**
- 高智商：能理解复杂逻辑
- 低记忆力：无法记住整个项目上下文
- 需要"认知"：通过结构化知识赋予 AI 项目理解能力

**解决方案：** 项目级知识沉淀，采用 MapReduce 架构

### 1.2 与简单 RAG 的区别

| 方面 | 简单 RAG | MapReduce 知识沉淀 |
|------|----------|-------------------|
| 输入 | 原始代码/文档 | 代码 + 专家规则 |
| 处理 | 分块 → 向量化 | Map 提取 → Merge → Reduce |
| 输出 | 语义相似的代码片段 | 高密度结构化知识 |
| 检索 | 语义匹配 | 分层精准定位 |
| 更新 | 重新索引 | 增量 Hash 更新 |

## 三、Map 阶段：代码结构提取

### 3.1 提取内容

对于每个 TypeScript 文件，提取以下结构化信息：

```typescript
interface FileInfo {
  filePath: string;                    // 文件路径
  moduleName: string;                  // 所属模块
  
  // 导出内容
  exports: {
    functions: FunctionInfo[];         // 导出函数
    classes: ClassInfo[];              // 导出类
    types: TypeInfo[];                 // 导出类型
    constants: ConstantInfo[];         // 导出常量
  };
  
  // 依赖关系
  dependencies: {
    internal: string[];                // 项目内部依赖
    external: string[];                // 外部依赖
  };
  
  // 代码结构
  structure: {
    lineCount: number;                 // 代码行数
    commentRatio: number;              // 注释比例
    complexity: number;                // 复杂度估算
  };
  
  // 业务语义（可选）
  semantics?: {
    purpose: string;                   // 功能描述
    relatedNodes?: string[];           // 关联的图节点
    relatedFlows?: string[];           // 关联的业务流程
  };
}
```

```

│  Map 阶段提取规则                                                         │
├─────────────────────────────────────────────────────────────────────────┤
│                                                                         │
│  【必须提取】                                                             │
│  ├── 导出的函数名和参数签名                                               │
│  ├── 导出的类名和方法列表                                                 │
│  ├── 导出的类型定义                                                       │
│  ├── import 语句中的依赖                                                  │
│  └── JSDoc 注释（如果有）                                                 │
```

## 四、Merge 阶段：模块级合并

### 4.1 合并策略

```
┌─────────────────────────────────────────────────────────────────────────┐
│  Merge 阶段流程                                                          │
├─────────────────────────────────────────────────────────────────────────┤
│                                                                         │
│  Step 1: 文件级 JSON 收集                                                │
│  ┌─────────────────────────────────────────────────────────────────┐   │
│  │  src/components/GraphEditor/*.json                              │   │
│  │  src/components/NodePanel/*.json                                │   │
│  │  src/components/PropertyPanel/*.json                            │   │
│  │  ...                                                             │   │
│  └─────────────────────────────────────────────────────────────────┘   │
│                              ↓                                          │
│  Step 2: 模块内 Merge（小粒度）                                          │
│  ┌─────────────────────────────────────────────────────────────────┐   │
│  │  合并同一目录下的所有 JSON                                        │   │
│  │  ├── 提取公共依赖                                                 │   │
│  │  ├── 合并导出 API 列表                                            │   │
│  │  ├── 建立文件间关系                                               │   │
│  │  └── 生成模块概览                                                 │   │
│  └─────────────────────────────────────────────────────────────────┘   │
│                              ↓                                          │
│  Step 3: 跨模块 Reduce（大粒度）                                         │
│  ┌─────────────────────────────────────────────────────────────────┐   │
│  │  合并所有模块的概览                                               │   │
│  │  ├── 生成架构图                                                  │   │
│  │  ├── 建立模块依赖关系图                                           │   │
│  │  ├── 生成 API 索引                                               │   │
│  │  └── 输出最终知识库                                               │   │
│  └─────────────────────────────────────────────────────────────────┘   │
│                                                                         │
└─────────────────────────────────────────────────────────────────────────┘
```

### 5.2 增量更新流程

```
┌─────────────────────────────────────────────────────────────────────────┐
│  增量更新流程                                                            │
├─────────────────────────────────────────────────────────────────────────┤
│                                                                         │
│  1. 扫描 src/ 目录                                                      │
│     ├── 获取所有 .ts/.tsx 文件                                          │
│     └── 排除 node_modules, test, dist                                   │
│                              ↓                                          │
│  2. 计算当前 Hash                                                       │
│     ├── 对每个文件计算 SHA256 Hash                                       │
│     └── 生成文件集合的总体 Hash                                          │
│                              ↓                                          │
│  3. 对比缓存 Hash                                                       │
│     ├── 未变化：跳过处理                                                 │
│     ├── 已变化：标记为需要重新 Map                                       │
│     ├── 新增：标记为需要 Map                                             │
│     └── 已删除：从缓存中移除                                             │
│                              ↓                                          │
│  4. 增量 Map 处理                                                       │
│     ├── 只处理变化的文件                                                 │
│     └── 更新对应的 JSON 输出                                             │
│                              ↓                                          │
│  5. 模块级 Hash 检测                                                    │
│     ├── 检查模块内是否有文件变化                                         │
│     └── 只有变化的模块需要重新 Merge                                     │
│                              ↓                                          │
│  6. 更新缓存                                                            │
│     ├── 保存新的 Hash 到缓存文件                                         │
│     └── 记录更新时间戳                                                   │
│                                                                         │
└─────────────────────────────────────────────────────────────────────────┘
```
