## 一、核心思路与理念

### 1.1 核心认知

> "AI 目前的价值还不在于一上来就可以独立投入工作。它只是一个拥有高智商、低记忆力的人。只是这样的人有无数个。"

**关键洞察：**
- AI 没有几十年的行业经验和适应一切的容错
- AI 会在一切该失败、该拒绝的地方失败或拒绝
- 错误构建的上下文信息、废弃的代码、不清晰的目标、无数内部沟通的专有名词，都会被 AI 无情地暴露出来

### 1.2 项目级知识沉淀的核心

> "项目级知识沉淀的核心，不是让 AI 记住一切，而是给它一个**稳定入口**，让它在解决问题前先读取项目事实。"

**为什么需要知识沉淀：**
- 在小项目中，AI 像在泳池中游泳，很快可以找到需要的内容
- 在大型项目（如腾讯游戏）中，AI 像在大海中游泳，需要人为指定方向和线索
- 服务个体的 AI 记忆架构可以运转，但服务整个项目的 AI 工作流无法只靠记忆系统（1M 上下文会被撑爆）

**结论：** 每天都是新的专家，AI 每次运行都要在大海里游很久找宝石，而且不知道有几颗宝石。

### 1.3 推荐读物

> "强烈推荐一本书：《Agentic Design Patterns》，由 Google 工程师 Antonio Gullí 所著。"

**核心概念：**
- Reflection（反思）
- Tool Use（工具使用）
- Planning（规划）
- Human-in-the-Loop（人在回路）
- Evaluation（评估）

> "吃透这些概念，已经足以应付绝大部分使用 AI 增效时会遇到的问题。"

## 二、具体做法详解

### 2.1 知识沉淀的内容选择

#### 高价值资产（值得沉淀）

| 资产类型 | 为什么值得沉淀 |
|----------|----------------|
| **代码** | 实际运行的最终产物，是逻辑的直接体现 |
| **测试用例** | 版本重复使用的可复用资产，是预期的行为验证 |

#### 低价值资产（不值得沉淀）

| 资产类型 | 为什么不值得沉淀 |
|----------|------------------|
| **Wiki 文档** | 大部分是人写的，覆盖面太小，很多内容已失去时效性，投入人力成本高 |
| **策划案** | 格式过多，没有统一路径；与实际执行有偏差和漂移；废弃的案子不会被删除 |
| **TAPD** | 人为产出的一次性产物，基本用完即弃，提炼价值风险高 |

### 2.2 MapReduce 架构详解

#### Map 阶段：单文件分析

**对于客户端 Lua 业务逻辑代码，提取的信息包括：**

```
┌─────────────────────────────────────────────────────────────┐
│  Map 阶段提取内容                                            │
├─────────────────────────────────────────────────────────────┤
│  • 类名、类功能                                               │
│  • 函数主要功能                                               │
│  • 涉及的模块                                                │
│  • 协议（Protobuf/网络协议）                                   │
│  • 配置表引用                                                │
│  • 枚举类型                                                  │
│  • 核心流程：                                                 │
│    - 资源加载                                                │
│    - 技能播放                                                │
│    - 网络通信                                                │
│    - 事件触发                                                │
│    - 关键日志                                                │
└─────────────────────────────────────────────────────────────┘
```

**对于服务器代码，关注：**
- 操作描述
- 状态转化
- 持久化/网络交互

**对于测试用例，提取：**
- 游戏流程
- 游戏对象识别
- 交互机制总结
- 功能特性归纳
- 边缘情况整理

#### Map 阶段输出格式

```json
{
  "file": "battle_system.lua",
  "classes": [
    {
      "name": "BattleManager",
      "function": "管理战斗流程的核心类",
      "methods": ["startBattle", "endBattle", "processTurn"],
      "protocols": ["proto_battle_start", "proto_battle_end"],
      "configTables": ["battle_config", "skill_config"],
      "enums": ["BattleState", "TurnType"],
      "coreFlows": {
        "resourceLoading": ["loadBattleUI", "initBattleData"],
        "networkComm": ["sendBattleAction", "recvBattleResult"],
        "events": ["onBattleStart", "onTurnChange", "onBattleEnd"]
      }
    }
  ]
}
```

#### Reduce 阶段：模块级合并

```
┌─────────────────────────────────────────────────────────────┐
│  Reduce 阶段流程                                             │
├─────────────────────────────────────────────────────────────┤
│                                                             │
│  单文件 JSON  ──→  模块内 Reduce ──→  模块级文档              │
│    ↓                    ↓                    ↓              │
│  file1.json          中间层合并            最终文档           │
│  file2.json          (分层Reduce)          (Markdown/JSON)   │
│  file3.json          解决信息密度丢失                         │
│  ...                                                            │
│                                                             │
└─────────────────────────────────────────────────────────────┘
```

#### Merge 与 Reduce 的详细实现

> 原文关键描述：
> "信息沉淀的最小子集应该是最小的模块子集。沉淀的形式我们采用的是 **MapReduce 架构**。"
> "自定义选择需要沉淀的模块子集，子集中的每个文件先进行一次 **Map 分析**，提炼出这个文件中对于自己业务有意义的东西。"
> "再将同一模块下的信息结构向上 **Merge** 得到模块级别的文档。"

**MapReduce 完整流程：**

```
┌─────────────────────────────────────────────────────────────────────────┐
│  MapReduce 知识沉淀架构                                                   │
├─────────────────────────────────────────────────────────────────────────┤
│                                                                         │
│  【Map 阶段】单文件分析                                                   │
│  ┌─────────────────────────────────────────────────────────────────┐   │
│  │  模块 A                                                          │   │
│  │  ├── file1.lua ──→ Map ──→ {类名, 功能, 协议, ...}              │   │
│  │  ├── file2.lua ──→ Map ──→ {类名, 功能, 协议, ...}              │   │
│  │  ├── file3.lua ──→ Map ──→ {类名, 功能, 协议, ...}              │   │
│  │  └── fileN.lua ──→ Map ──→ {类名, 功能, 协议, ...}              │   │
│  └─────────────────────────────────────────────────────────────────┘   │
│                              ↓                                          │
│  【Merge 阶段】模块内合并                                                 │
│  ┌─────────────────────────────────────────────────────────────────┐   │
│  │  模块 A 的所有 JSON ──→ Merge ──→ 模块 A 级别文档                  │   │
│  │  (分层 Merge，解决信息密度丢失)                                    │   │
│  └─────────────────────────────────────────────────────────────────┘   │
│                              ↓                                          │
│  【Reduce 阶段】跨模块汇总                                               │
│  ┌─────────────────────────────────────────────────────────────────┐   │
│  │  模块 A 文档 + 模块 B 文档 + ... ──→ Reduce ──→ 知识库最终文档       │   │
│  └─────────────────────────────────────────────────────────────────┘   │
│                                                                         │
└─────────────────────────────────────────────────────────────────────────┘
```

**Merge 的关键实现细节：**

1. **分层 Merge 策略**
   - 先在模块内进行小粒度 Merge
   - 再进行跨模块的大粒度 Merge
   - 避免一次性处理过多文件导致信息密度丢失

2. **Merge 时的信息整合**
   ```json
   {
     "module": "battle_system",
     "files": ["battle_manager.lua", "battle_ui.lua", "battle_network.lua"],
     "classes": [
       // 合并所有文件的类信息
     ],
     "protocols": [
       // 合并所有文件的协议引用
     ],
     "configTables": [
       // 合并所有文件的配置表引用
     ],
     "coreFlows": {
       // 整合完整的业务流程
     },
     "dependencies": {
       // 模块间依赖关系
     }
   }
   ```

3. **Merge 时的去重与关联**
   - 识别跨文件的类引用关系
   - 合并相同协议的不同使用场景
   - 建立配置表之间的关联

#### 遇到的挑战及解决方案（原文详细描述）

> "对于单文件的 Map 分析要点，是我们一次又一次打磨出来的需求点。Map 的过程中，**单文件上下文溢出时怎么裁剪比较合理**；Map 阶段没有文件间的关系信息，**Reduce 时高达上千文件的模块即使分层 Reduce 也会产生信息密度丢失的问题**；MapReduce 架构随着**测试用例的线性增长开始显现压力**；怎么实现**文件级/模块级 Hash 缓存实现增量更新**。这些问题都是需要在实际沉淀中去打磨出来的。"

| 挑战 | 详细描述 | 解决方案 |
|------|----------|----------|
| **单文件上下文溢出** | 大文件超过模型上下文限制 | 合理裁剪上下文，只保留核心信息 |
| **Reduce 信息密度丢失** | 千文件模块合并后信息稀疏 | 分层 Merge，先模块内后跨模块 |
| **测试用例线性增长压力** | 用例数量增加导致处理时间增长 | 优化索引，分批处理 |
| **增量更新** | 全量重新处理成本过高 | 文件级/模块级 Hash 缓存 |

#### Hash 缓存增量更新机制

```
┌─────────────────────────────────────────────────────────────────────────┐
│  增量更新流程                                                             │
├─────────────────────────────────────────────────────────────────────────┤
│                                                                         │
│  1. 计算每个文件的 Hash                                                   │
│     ┌─────────────────────────────────────────────────────────────┐    │
│     │  file1.lua → hash: abc123 (未变化) → 跳过                     │    │
│     │  file2.lua → hash: def456 (未变化) → 跳过                     │    │
│     │  file3.lua → hash: ghi789 (已变化)  → 重新 Map                │    │
│     │  file4.lua → hash: NEW (新文件)     → 新增 Map                │    │
│     │  file5.lua → hash: MISSING (已删除) → 移除对应文档             │    │
│     └─────────────────────────────────────────────────────────────┘    │
│                              ↓                                          │
│  2. 只处理变更的文件                                                       │
│     ┌─────────────────────────────────────────────────────────────┐    │
│     │  file3.lua ──→ 重新 Map ──→ 新的 JSON 结构                   │    │
│     └─────────────────────────────────────────────────────────────┘    │
│                              ↓                                          │
│  3. 模块级 Hash 检测                                                      │
│     ┌─────────────────────────────────────────────────────────────┐    │
│     │  模块 A 的文件集合 Hash 变化 → 重新 Merge 模块 A              │    │
│     │  模块 B 的文件集合 Hash 未变 → 跳过模块 B                      │    │
│     └─────────────────────────────────────────────────────────────┘    │
│                              ↓                                          │
│  4. 最终 Reduce 输出                                                      │
│     ┌─────────────────────────────────────────────────────────────┐    │
│     │  只有变更的模块参与最终 Reduce → 更新知识库                    │    │
│     └─────────────────────────────────────────────────────────────┘    │
│                                                                         │
└─────────────────────────────────────────────────────────────────────────┘
```

**Hash 缓存存储结构：**
```json
{
  "moduleHashes": {
    "battle_system": "module_hash_v2",
    "ui_system": "module_hash_v1"
  },
  "fileHashes": {
    "battle_system": {
      "battle_manager.lua": "abc123",
      "battle_ui.lua": "def456"
    }
  },
  "lastUpdate": "2026-06-22T10:00:00Z"
}
```

#### 知识库的组织与索引

> 评论区关键信息（来自 junyiwwu 的问题和 poanshen 的回答）：
> "文档粒度看文章似乎已经细化到具体的单文件类了，那意味着整个工程里文档是非常庞大的。是如何组织和控制 agent 的索引效率呢？"
> 
> "我们把AI总结归纳出来的内容放到RAG里面做成mcp给agent使用。...在我们查bug的工作流中我们会**提示AI第一步先查知识库**，根据知识库里面返回的内容找到可能的代码文件的关键词，然后直接读代码。在我们的应用中RAG能够避免agent像大海捞针一样的翻代码库。"

**知识库索引策略：**

1. **RAG 向量化存储**
   - 将 Merge 后的模块级文档向量化
   - 存储到向量数据库
   - 封装成 MCP 接口供 Agent 调用

2. **分层检索**
   ```
   Agent 收到问题
        ↓
   第一步：查询知识库（RAG）
        ↓
   获取：相关模块、协议、配置表、流程信息
        ↓
   第二步：根据知识库返回的关键词定位代码文件
        ↓
   第三步：读取具体代码进行分析
   ```

3. **避免大海捞针**
   - 知识库提供方向和线索
   - 不是直接让 Agent 搜索整个代码库
   - 先从知识库获取关键词，再精准读取代码

### 2.5 验证知识沉淀的有效性

> "知识沉淀必须用**真实问题验证**，不能只看检索结果是否看起来相关。"

**验证方法：**
- 准备 100 个标准问答
- 涵盖客户端实现技术、游戏游玩细节
- 100% 通过才算达标

**示例问题：**
> "某关卡中，content ****/**** 已通过 DelRefreshContentNpc 删除，为什么玩家移动触发 GridChanged 后又被 AOI 重新刷出？"

**AI 分析流程（四步定位）：**
1. 定位配置链
2. 定位删除链
3. 定位重刷链
4. 给出根因

**结论：** `DelRefreshContentNpc` 删除了运行时 content/NPC 实例，但没有屏蔽静态配置层；玩家移动触发 AOI 区域加载时，系统又按静态配置把它补回来了。

## 三、AI 增效场景的落地

### 3.1 三个维度判断是否要 AI 增效

| 维度 | 评估内容 |
|------|----------|
| **收益 vs 成本** | AI 增效后的实际收益有多少？迁移到这一套架构的成本是否超过对应收益？是否存在迁移风险？ |
| **普适性** | 对应的普适性有多少？有多少其他业务可以因为这套架构受益？ |
| **SOP 流程** | 是否已经有一套标准的 SOP 流程？流程中 AI 可以代替的环节是否清晰明确？ |

> "很多时候我们拿到一个 AI 增效的场景，就只能看到场景的起点和终点，然后就想着 AI 带着我们直接从起点走向终点。这是荒谬的。"

**正确做法：**
1. 先仔细构思 AI 出现前全人工流程的拆解
2. 将其中收益最大的部分思考用 AI 替换
3. 评估替换后对上下游环节的影响
4. 缺失工具和能力的部分，制作原子化的工具给到 AI

### 3.2 原子能力形态

#### MCP 工具示例

**配置表读写 MCP：**
- 按表名/描述/别名找到目标表
- 按主键定位某一行
- 按字段路径批量取值
- 全局关键字搜索

**日志分析 MCP：**
- 登录链路分析
- 进入场景（首次进入/切换/传送/副本/断线重连的完整状态机）
- FSM 状态流转
- 任务、对话、NPC 交互
- Lua 异常检测

**物理场景查询 CLI：**
- 直接读取 UE 导出的 PhysX 二进制场景数据
- 不启动引擎完成空间查询
- 用于种怪、投放采集物、摆放 NPC 和交互物

**其他 MCP：**
- UAsset 资产分析
- UTrace 性能分析
- Lua Runtime 连接 UE 执行

#### 实际效果对比

| 场景 | AI 之前 | AI 之后 |
|------|---------|---------|
| **配置对话任务** | 30分钟（8张表，560列） | **2分钟** |
| **配置任务链** | 几天 | **几小时** |

> "原本做不到，是因为 AI 没有稳定入口去理解项目事实；现在可以做，是因为这些事实被沉淀成了可检索、可组合、可持续更新的知识接口。"

### 3.3 用专家经验指引 AI 行动

> "每个人的工作经验、工作习惯，其实都有变化和不变的部分。我们执行不变的流程，遇到变量就进入新的流程。而这些不变的流程就是 AI 的行动方针。"

**SOP 流程构建要点：**

1. **抽象成 SOP 流程**
   - 如果自己口头都说不清在做什么，AI 更学不会

2. **自行验证 SOP**
   - 很多细节自己并未察觉到，没有写进流程本身
   - 人天然拥有更强的纠错能力，很难踩第二个一模一样的坑
   - 但 AI 会再踩一次

3. **AI 替换机械节点**
   - 一个好的 SOP 流程，一定可以让 AI 替换掉其中大部分机械且稳定的节点
   - 从执行者变为指挥者

## 四、共建 AI 生态

### 4.1 提供通用原子能力

> "原子能力说得很高尚，其实也就是一些复用性较高的脚本和接口罢了。"

**做法：**
- 鼓励项目中的同学积极沉淀自己日常工作中最常用的重复性操作和脚本
- 从 AI 的角度替他们评估可行性和稳定性

### 4.2 与业务一线合作

> "越优秀的一线同学，越不是凭空灵感在工作，而是有一套稳定的判断顺序。"

**合作模式：**
1. 不是告诉业务同学"AI 可以做什么"
2. 而是业务同学告诉我们"这件事平时到底是怎么做的"
3. 把这套判断顺序变成可以讨论、可以试用、可以继续打磨的业务工具

**案例：活动策划合作**
- 活动创意生成：先对齐活动目标、用户层级、版本节奏、奖励投放、参与门槛、复用资源，再生成创意
- 活动需求落地：拆到规则说明、配置需求、界面需求、埋点需求、测试关注点和风险点

### 4.3 搭建共享平台

> "很多 AI 工具停留在脚本、Prompt、Skill 里，用的人知道怎么跑，不用的人完全不知道它是什么。"

**NRC_AI 共享平台功能：**
- 应用可以搜索、浏览、安装、评分、评论
- 通过项目、草稿、版本发布流程进行协作维护
- 每个应用不只是一个 Prompt，而是一套可以带着 Skill、表单、对话模式、工作目录和运行配置的业务场景
- Widget 形态：表单式、持续对话、一键执行、网页展示

## 五、团队自身优化

### 5.1 低成本摆脱模型依赖

> "一开始我们使用了司内部分公共平台的 API，后来发现并发量不够，服务稳定性不强，上下文限制太大。"

**解决方案：**
- 使用 Mac 的 MLX 架构作为主要的模型能力来源
- 在 Mac 平台上低成本部署多个 Qwen 中小模型
- 使用 TurboQuant KV Cache、SpecPrefill、混合精度、MTP 等优化手段
- 最终达到：单 Mac 机器 700 tokens/s 的 prefill 和 70 tokens/s 的输出速度

**模型使用策略：**

| 场景 | 模型选择 |
|------|----------|
| 知识沉淀、批量总结、日志初筛、Bug 定位 | 本地模型 |
| 高难推理、关键评估、需要更强泛化能力 | 更强模型 |

> "AI 到了下半场，模型应该是一组可以调度的资源，而不是一个唯一答案。"

### 5.2 把视线回到传统工作流质量

> "AI 没有几十年的行业经验和适应一切的容错。AI 会在一切该失败、该拒绝的地方失败或拒绝。"

**AI 暴露的问题：**

| 问题类型 | 表现 | 后果 |
|----------|------|------|
| 日志问题 | 部分日志级别过高、过于繁琐 | AI 严重误判 |
| Bug 单描述不清 | 缺少复现路径 | AI 只能猜测 |
| 测试用例命名随意 | 难以理解预期行为 | AI 难以覆盖 |
| 内部黑话过多 | 团队默契当成缺失信息 | AI 理解偏差 |

> "AI 像一面没有情面的镜子，照出来的不只是模型能力，也照出来我们工作流里的灰尘。"

> "AI 用不好，很多时候不是 AI 不够聪明，而是我们给它的世界本身就是混乱的。"

## 六、AI Bug 修复回测专项

### 6.1 为什么需要回测

> "如果没有回测，AI 增效很容易只剩下体感。某次演示效果很好，某个样例结果很惊艳，都不能说明它真的稳定可用。"

### 6.2 回测流程

```
┌─────────────────────────────────────────────────────────────┐
│  Bug 回测流程                                                │
├─────────────────────────────────────────────────────────────┤
│                                                             │
│  TAPD Webhook ──→ 过滤去重 ──→ 任务队列 ──→ Worker 处理     │
│       │           │            │            │               │
│   真实Bug      状态合适       等待处理     更新代码          │
│   有日志证据   有日志证据                  读取日志          │
│                                          调用AI分析         │
│                                                ↓           │
│                                          分析结果回写        │
│                                                ↓           │
│                                    Bug 进入已解决状态       │
│                                                ↓           │
│                                      进入评估候选           │
│                                                ↓           │
│                              比较 AI 分析 vs 实际修复        │
│                                                ↓           │
│                              评估：correct/partially_correct/wrong/unjudgeable
│                                                             │
└─────────────────────────────────────────────────────────────┘
```

### 6.3 评估维度

| 维度 | 评估内容 |
|------|----------|
| 根因是否吻合 | AI 判断的根因 vs 实际修复的根因 |
| 模块/文件是否定位准确 | AI 定位的位置 vs 实际修改的位置 |
| 日志解释是否合理 | AI 对日志的解读是否正确 |
| 修复建议是否有参考价值 | AI 的建议是否对实际修复有帮助 |

### 6.4 问题暴露与迭代

> "我们想要的就是不断暴露问题。"

| 问题类型 | 说明 | 解决方向 |
|----------|------|----------|
| 某类日志 AI 经常误解 | 说明日志分析 MCP 的规则不够 | 补充 MCP 规则 |
| 某模块经常定位偏移 | 说明知识沉淀或模块索引需补 | 补充知识库 |
| 某类 Bug 分析完整但修复建议无用 | 说明 Skill 停留在"解释现象" | 收紧 Skill 流程 |

> "判断标准不应该是'这个东西是不是新'，也不应该是'这个东西是不是我们自己做的'，而是它有没有进入真实工作流，有没有通过回测，有没有持续产生收益。"

## 七、核心总结

### 7.1 金句摘录

> "AI 的边界之外，仍然是人的判断、人的责任、人的创造力。"

> "Think what AI can't. Build what AI can."

> "不能进入工作流的工具，再精巧也只是玩具；不能被验证的优化，再漂亮也只是感觉。"

> "AI 增效还是回到人本身。AI 可以放大流程、复制经验、执行工具，但方向从哪里来、边界怎么定、结果是否可信，仍然依赖人的判断。"

### 7.2 关键方法论

| 方法论 | 具体做法 |
|--------|----------|
| **知识沉淀** | MapReduce 架构，从代码和测试用例提取高密度信息 |
| **Tool Use** | MCP/CLI 扩展 AI 能力边界 |
| **Skill 沉淀** | 用专家经验指引 AI 行动 |
| **Evaluation** | 清晰的回测机制持续迭代 |
| **做加法** | 保持敏感，敢于拿真实业务去试新技术 |
| **做减法** | 保持清醒，砍掉过时的流程和技术 |

### 7.3 对我们项目的启示

1. **知识沉淀粒度** - 不是简单 RAG，而是先 Map 提取结构化信息，再 Merge 模块内合并，最后 Reduce 跨模块汇总
2. **分层处理策略** - 先小粒度 Merge（模块内），再大粒度 Reduce（跨模块），避免信息密度丢失
3. **增量更新机制** - Hash 缓存实现只处理变更部分（文件级 + 模块级）
4. **高密度信息** - JSON 结构比原始代码更适合 AI 理解
5. **每日自动更新** - 完全自动化，无人工参与
6. **分层检索策略** - 先查知识库获取方向，再根据关键词精准读取代码，避免大海捞针
7. **回测验证** - 不能只看体感，要有系统化的评估机制
8. **工作流质量** - AI 会暴露文档、日志、需求中的问题，要持续治理

### 7.4 原文关键原文摘录

**关于 MapReduce 的原文描述：**
> "信息沉淀的最小子集应该是最小的模块子集。沉淀的形式我们采用的是 MapReduce 架构。自定义选择需要沉淀的模块子集，子集中的每个文件先进行一次 Map 分析，提炼出这个文件中对于自己业务有意义的东西。"

**关于 Merge 的原文描述：**
> "再将同一模块下的信息结构向上 Merge 得到模块级别的文档。"

**关于挑战的原文描述：**
> "Map 的过程中，单文件上下文溢出时怎么裁剪比较合理；Map 阶段没有文件间的关系信息，Reduce 时高达上千文件的模块即使分层 Reduce 也会产生信息密度丢失的问题；MapReduce 架构随着测试用例的线性增长开始显现压力；怎么实现文件级/模块级 Hash 缓存实现增量更新。这些问题都是需要在实际沉淀中去打磨出来的。"

**关于知识库使用的原文描述：**
> "在我们查bug的工作流中我们会提示AI第一步先查知识库，根据知识库里面返回的内容找到可能的代码文件的关键词，然后直接读代码。在我们的应用中RAG能够避免agent像大海捞针一样的翻代码库。"
